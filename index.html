Java动态代理机制分析

發表於 2015-09-24   |   分類於 java

引言

前几天学习Spring的AOP。其实质也是代理机制，想着更加深入的去了解一下这方面的内容。本文不是原创内容，是在原有文章上面简化抽出自己所需要的部分。如果想要了解原文，请点击Java 动态代理机制分析及扩展。


首先我们先了解一下什么是代理。如上图所示。其实代理模式就是具体的操作让委托类去完成，而代理类负责传话，这样就避免了委托类与访问者直接的接触。从设计上得到更大的灵活性。

相关的类和接口
java.lang.reflect.Proxy：这是 Java 动态代理机制的主类，它提供了一组静态方法来为一组接口动态地生成代理类及其对象。

清单 1. Proxy 的静态方法

// 方法 1: 该方法用于获取指定代理对象所关联的调用处理器 static InvocationHandler getInvocationHandler(Object proxy) // 方法 2：该方法用于获取关联于指定类装载器和一组接口的动态代理类的类对象 static Class getProxyClass(ClassLoader loader, Class[] interfaces) // 方法 3：该方法用于判断指定类对象是否是一个动态代理类 static boolean isProxyClass(Class cl) // 方法 4：该方法用于为指定类装载器、一组接口及调用处理器生成动态代理类实例（最常用） //第一个参数是一个ClassLoader对象，定义了由哪个ClassLoader对象来对生成的代理对象进行加载； //第二个参数是一个Interface对象的数组，表示的是我将要给我需要代理的对象提供一组什么接口，如果我提供了一组接口给它，那么这个代理对象就宣称实现了该接口(多态)，这样我就能调用这组接口中的方法了； //第三个参数是一个InvocationHandler对象，表示的是当我这个动态代理对象在调用方法的时候，会关联到哪一个InvocationHandler对象上 static Object newProxyInstance(ClassLoader loader, Class[] interfaces, InvocationHandler h)
java.lang.reflect.InvocationHandler：这是调用处理器接口，它自定义了一个 invoke 方法，用于集中处理在动态代理类对象上的方法调用，通常在该方法中实现对委托类的代理访问。

清单 2. InvocationHandler 的核心方法
`// 该方法负责集中处理动态代理类上的所有方法调用。第一个参数既是代理类实例，第二个参数是被调用的方法对象
// 第三个方法是调用参数。调用处理器根据这三个参数进行预处理或分派到委托类实例上发射执行
Object invoke(Object proxy, Method method, Object[] args)`

代理机制及其特点

首先让我们来了解一下如何使用 Java 动态代理。具体有如下四步骤：
1 通过实现 InvocationHandler 接口创建自己的调用处理器； 2 通过为 Proxy 类指定 ClassLoader 对象和一组 interface 来创建动态代理类；
3 通过反射机制获得动态代理类的构造函数，其唯一参数类型是调用处理器接口类型； 4 通过构造函数创建动态代理类实例，构造时调用处理器对象作为参数被传入。
介绍完这个两个接口（类）之后，通过一个实例来具体看看动态代理的。

首先我们定义一个Subject的接口。定义两个方法：

`public interface Subject
{
public void send();
public void hello(String str);
}`

接着我们用一个类来实现这个接口，类的名字就叫做RealSubject

`public class RealSubject implements Subject
{
    @Override
    public void rent()
    {
        System.out.println("I want to rent my house");
    }

    @Override
    public void hello(String str)
    {
        System.out.println("hello: " + str);
    }
}`

下一步，我们就要定义一个动态代理类了，前面说个，每一个动态代理类都必须要实现 InvocationHandler 这个接口，因此我们这个动态代理类也不例外：

`public class DynamicProxy implements InvocationHandler
{
    //　这个就是我们要代理的真实对象
    private Object subject;

    //    构造方法，给我们要代理的真实对象赋初值
    public DynamicProxy(Object subject)
    {
        this.subject = subject;
    }

    @Override
    public Object invoke(Object object, Method method, Object[] args)
            throws Throwable
    {
        //　　在代理真实对象前我们可以添加一些自己的操作
        System.out.println("before rent house");

        System.out.println("Method:" + method);

        //    当代理对象调用真实对象的方法时，其会自动的跳转到代理对象关联的handler对象的invoke方法来进行调用
        method.invoke(subject, args);

        //　　在代理真实对象后我们也可以添加一些自己的操作
        System.out.println("after rent house");

        return null;
    }

}`

最后再看看客户端：

`public class Client
{
    public static void main(String[] args)
    {
        //    我们要代理的真实对象
        Subject realSubject = new RealSubject();

        //    我们要代理哪个真实对象，就将该对象传进去，最后是通过该真实对象来调用其方法的
        InvocationHandler handler = new DynamicProxy(realSubject);

        /*
         * 通过Proxy的newProxyInstance方法来创建我们的代理对象，我们来看看其三个参数
         * 第一个参数 handler.getClass().getClassLoader() ，我们这里使用handler这个类的ClassLoader对象来加载我们的代理对象
         * 第二个参数realSubject.getClass().getInterfaces()，我们这里为代理对象提供的接口是真实对象所实行的接口，表示我要代理的是该真实对象，这样我就能调用这组接口中的方法了
         * 第三个参数handler， 我们这里将这个代理对象关联到了上方的 InvocationHandler 这个对象上
         */
        Subject subject = (Subject)Proxy.newProxyInstance(handler.getClass().getClassLoader(), realSubject
                .getClass().getInterfaces(), handler);

        System.out.println(subject.getClass().getName());
        subject.rent();
        subject.hello("world");
    }
}`

生成结果：

`$Proxy0
before rent house
Method:public void com.cglib.RealSubject.rent()
I want to rent my house
after rent house
before rent house
Method:public void com.cglib.RealSubject.hello(java.lang.String)
hello: world
after rent house`

如果是没有接口呢，我们可以用cglib来实现代理。他的原理是对指定的目标类生成一个子类，并覆盖其中方法实现增强，但因为采用的是继承，所以不能对final修饰的类进行代理。
创建一个类：

`public class RealSubject {
    public void rent()
    {
        System.out.println("I want to rent my house");
    }
    public void hello(String str)
    {
        System.out.println("hello: " + str);
    }
}`

实现代理类：

`public class DynamicProxy implements MethodInterceptor {

    private Object target;  
    /** 
     * 创建代理对象 
     *  
     * @param target 
     * @return 
     */  
    public Object getInstance(Object target) { 
        this.target = target;  
        Enhancer enhancer = new Enhancer();  
        enhancer.setSuperclass(this.target.getClass());  
        // 回调方法  
        enhancer.setCallback(this);  
        // 创建代理对象  
        return enhancer.create();  
    }  

    // 回调方法  
    public Object intercept(Object obj, Method method, Object[] args,  
            MethodProxy proxy) throws Throwable {  
        //　　在代理真实对象前我们可以添加一些自己的操作
        System.out.println("before rent house");

        System.out.println("Method:" + method); 
        proxy.invokeSuper(obj, args);  
        //　　在代理真实对象后我们也可以添加一些自己的操作
        System.out.println("after rent house");
        return null;  


    }  
}`

客户端的方法：

`public class Client {
    public static void main(String[] args) {  
        DynamicProxy cglib=new DynamicProxy();  
        RealSubject bookCglib=(RealSubject)cglib.getInstance(new RealSubject()); 
        System.out.println(bookCglib.getClass().getName());
        bookCglib.rent();
        bookCglib.hello("world");
    }  
}`

生成结果：

`com.cglib.RealSubject$$EnhancerByCGLIB$$799f4d5a
before rent house
Method:public void com.cglib.RealSubject.rent()
I want to rent my house
after rent house
before rent house
Method:public void com.cglib.RealSubject.hello(java.lang.String)
hello: world
after rent house`

最后我们的生成对象是个RealSubject的实例，而不是一个由jvm生成的代理对象。这个跟用jdk使用是有区别的。以$开头，proxy为中，最后一个数字表示对象的标号。

ORA-01810的错误

發表於 2015-09-18   |   分類於 数据库

今天在写一个查询的时候用到to_date方法，我用

to_date('2015-5-1 00:00:00','yyyy-MM-dd HH:mm:ss')
结果报ORA-01810 “格式代码出现两次”的错误.后来上网查询后,发现是格式化字符串有问题,其中不应该将某个格式重复两次,否则Oracle就不知道要从哪个占位区间去解析该字段了,正确的格式应该用MI来代表分钟,而不是java中的ss
Sql代码

to_date('2015-5-1 00:00:00','yyyy-MM-dd HH:MI:SS')
24小时的可以用HH24来代替HH.

实施部署

發表於 2015-09-01   |   分類於 java

在部署系统的时候，oracle是在集群上面。所以我们原先用使用sid来配置jdbcurl已经不行了，会导致服务启动不起来，因为对于集群来讲，每一个节点的sid都是不一样的。所以我们将原先的格式修改为集群下的配置，将sid来替换为service_name ，service_name 包含所有的节点。
原来的配置：
<Set name="jdbcUrl">jdbc:oracle:thin:@172.16.2.69:1521:ahltollglc</Set>
修改为：
<Set name="jdbcUrl">jdbc:oracle:thin:@//172.16.2.69:1521/ahltollglc</Set>
注：@ 后面加上//这是与使用sid的主要区别。
其实oracle连接java jdbc/ojdbc 驱动的方式一共有三种。第一种是Oracle Thin JDBC Driver，第二种是Oracle OCI JDBC Driver，第三种是Oracle 8i OCI JDBC Driver 。上面说的那两种配置的方式其实都是使用的Oracel Thin JDBC Driver这种驱动的方式，这种还有一种形式就是用数据库的TNSName来配置，如：
jdbc:oracle:thin:@TNSName
注：TNSName是本地的名称，这种方式我没有进行配置，因为这是本地命名服务配置的。估计远程的话这种方式应该是连接不上的，但是没有经过测试。
像Oracle OCI JDBC Driver这种方式配置url的方式为：
jdbc:oracle:oci:@port/service_name
//或者
jdbc:oracle:oci:@port:sid
oci这种形式是一种胖客户端的连接方式，而thin是一种瘦客户端的形式。用oci的话需要本地安装oracle的客户端，thin的话就不需要，所以大多数都是用的thin这种连接方式。最后还有一种是8i OCI这种驱动连接的方式，配置的方式为：
jdbc:oracle:oci8:@TNSName
这种需要引进classes12.zip这个文件。
以上就是在工作中总结的oracle的几种连接的方式。推荐的还是用service_name这种形式，官方现在也是推荐用这种形式，不再用sid这种去配置url。

关于spring配置文件中resourceRef的配置

發表於 2015-08-27   |   分類於 java

关于spring配置文件中resourceRef的配置。收费站和管理处的程序，有些是部署在tomcat，有些是部署在weblogic上面。但是在不同的容器中resourceRef的配置是不同的，我们在代码中默认是false，但在tomcat上面要设置true，这样才能保证服务能够正常的启动。
在tomcat上面：
<bean id="dataSource" class="org.springframework.jndi.JndiObjectFactoryBean">
<property name="resourceRef">
<value>true</value>
</property>
<property name="jndiName">
<value>jdbc/toll</value>
</property>
</bean>
我查了一下JndiObjectFactoryBean中的关于resourceRef的描述，在JndiLocatorSupport的抽象类中有这样的解释:

/**
* Set whether the lookup occurs in a J2EE container, i.e. if the prefix
* "java:comp/env/" needs to be added if the JNDI name doesn't already
* contain it. Default is "false".
* <p>Note: Will only get applied if no other scheme (e.g. "java:") is given.
*/

所以在tomcat中设置jndi的时候需要加上java:comp/env/，因此要设置为true，因为设置成为true之后，程序会自动添加java:comp/env/给jndi。设置为true对weblogic，jetty这些容器也不会产生影响，这样的话，我们在下收费站实施部署的时候，就不用考虑容器是什么而去修改这些配置文件了。

jndi多数据源配置

發表於 2015-08-21   |   分類於 java

今天早上跟领导讨论下关于报表服务启动一个，同时由稽查和收费两个系统访问的问题。这样就省的再启动一个多余的report服务，节省了服务器资源。所以我用中午的时间做了一下测试，验证了一下。其实这个主要的问题还是在于是否支持多数据源，因为稽查和收费不在同一个数据库中。查看下fineReport的api，这个工具是能够同时支持多个数据源的。然后再看spring,spring对多数据源也能够很好的支持。所以问题就迎刃而解了。下面是测试的步骤：
1.配置applicationContext.xml。

<!-- 收费-->
<bean id="dataSource" class="org.springframework.jndi.JndiObjectFactoryBean">
<property name="resourceRef">
<value>true</value>
</property>
<property name="jndiName">
<value>jdbc/toll</value>
</property>
</bean>
<!-- 稽查-->
<bean id="auditDataSource" class="org.springframework.jndi.JndiObjectFactoryBean">
<property name="resourceRef">
<value>true</value>
</property>
<property name="jndiName">
<value>jdbc/audit</value>
</property>
</bean>

这里面report项目中并不需要jdbctemper之类的，所以不需要来引用datasource。如果在别的项目中的话，需要重写下AbstractRoutingDataSource的determineCurrentLookupKey方法来获得不同的数据源。这个类的父类AbstractDataSource是spring获得数据源的类。今天只是测试report的服务，所以没再继续看这块。

这里只需要切换一下，在报表模板文件中的datasource的名字。如我们将稽查的数据源的名字定义为auditDataSource。然后只需要将模板中的DatabaseName改为这个名字即可。如：
